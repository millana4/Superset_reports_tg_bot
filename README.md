# Телеграм-бот для рассылки отчётов из Superset

## Назначение
Бот автоматически пересылает отчёты с BI-платформы Apache Superset в Telegram — отдельным
пользователям и в группы — через технические email-ящики. Он реагирует на входящие письма
с вложениями (PDF, PNG, JPG, JPEG) и доставляет их подписчикам.

## Как работает бот

### Регистрация пользователей
- Системный администратор добавляет пользователя в таблицу NocoDB: имя, телефон и ящики,
  с которых пользователь будет получать отчёты.
- Пользователь запускает бота и делится своим номером телефона.
- Бот обращается по API к NocoDB и проверяет, есть ли пользователь с таким телефоном.
  Если есть — записывает его `id_telegram`, по которому потом идёт рассылка.
- Телефоны нормализуются «лениво»: при первой авторизации телефон приводится к единому
  формату (`+7XXXXXXXXXX`) и в этом виде перезаписывается обратно в таблицу.
- Регистрация завершена — пользователь получает подтверждение, что подписан на рассылку.

### Регистрация групп (чатов)
- Системный администратор добавляет название группы и ящики, с которых в группу нужно
  пересылать вложения. Названия групп **не уникальны**.
- Администратор группы добавляет бота в группу и выдаёт ему права администратора.
- По событию получения прав администратора бот обращается к NocoDB, находит группу по
  названию и записывает её `id_telegram_chat`.
- После записи группа помечается флагом `is_locked`, который защищает её `id_telegram_chat`
  от перезаписи. Это не позволяет второй группе с тем же названием перехватить рассылку.

### Рассылка
- Бот «слушает» технические ящики через IMAP в режиме IDLE — каждый ящик в отдельном потоке.
- Superset отправляет отчёты на эти ящики.
- При получении письма с вложением бот:
  - извлекает тему и вложения допустимых форматов (PDF, PNG, JPG, JPEG); текстовые части
    письма игнорируются;
  - находит в NocoDB получателей, подписанных на этот ящик, — пользователей и группы;
  - рассылает отчёт по списку `id_telegram`.

## Основные компоненты

**NocoDB** — конфигурационная база. Хранит пользователей, группы, почтовые ящики и связи
между ними (какие пользователи и группы подписаны на какие ящики). Доступ — по REST API.

**IMAP Idle Listener** — слушает входящие письма на почтовых ящиках через IMAP IDLE.
Каждый ящик обрабатывается в отдельном потоке.

**Telegram Bot** — отвечает на команды пользователей (старт, авторизация по контакту),
регистрирует группы и рассылает вложения из писем.

## Структура проекта
- `main.py` — точка входа: инициализация логирования, запуск бота и IMAP-слушателей.
- `bot.py` — создание экземпляра бота (с поддержкой прокси для отладки).
- `config.py` — чтение настроек из переменных окружения.
- `db.py` — слой доступа к данным NocoDB (пользователи, чаты, ящики).
- `nocodb_client.py` — низкоуровневый асинхронный клиент NocoDB REST API.
- `email_handler.py` — приём писем по IMAP, разбор вложений и рассылка.
- `handlers.py` — обработчики команд пользователя (`/start`, авторизация по контакту).
- `telegram_api.py` — обработка событий добавления бота в группу.
- `keyboards.py` — клавиатуры Telegram.
- `utils.py` — вспомогательные функции (нормализация телефона).
- `custom_logging.py` — настройка логирования и маскирование персональных данных.

## Настройка окружения
Переменные задаются в файле `.env` (см. `env.example`):

- `BOT_TOKEN` — токен Telegram-бота.
- `BOT_PROXY` — прокси для отладки. На сервере оставить пустым.
- `IMAP_SERVER`, `IMAP_EMAIL_SR0X`, `IMAP_PASSWORD_SR0X` — параметры технических ящиков.
- `NOCODB_SERVER`, `NOCODB_API_TOKEN` — адрес и токен NocoDB.
- `NOCODB_USERS_TABLE_ID`, `NOCODB_MAILBOXES_TABLE_ID`, `NOCODB_CHATS_TABLE_ID` — ID таблиц.
- `LOG_LEVEL` — уровень логирования (`INFO` на проде, `DEBUG` для отладки).

## Логирование
- Логи пишутся в `logs/bot.log` с ротацией (5 МБ, до 3 архивов) и сразу сбрасываются в файл.
- Персональные данные (телефоны, имена) маскируются и выводятся только на уровне `DEBUG`.
  На уровне `INFO` и выше персональные данные в логи не попадают.

## Запуск в Docker на сервере
Требуется установленный Docker и Docker Compose. Настройки берутся из `.env`
(см. `env.example`).

Сборка и запуск в фоне:

```
docker compose up -d --build
```

Контейнер запускается с политикой `restart: unless-stopped`: он автоматически
перезапускается при сбое и поднимается после перезагрузки сервера. После ручной
остановки (`docker compose down`) сам не стартует.
